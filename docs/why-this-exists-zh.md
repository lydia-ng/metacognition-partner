# 这份 skill 为什么存在

## 一个真实的起源

这份 skill 不是先有理论再写的产物。它是从一次具体的、长时间的复杂认知工作中**反向抽取**的。

具体来说——一次需要长时间、多轮、跨越多个分析框架的人际分析。在做这件事的过程中，我反复识别出自己的几个动作：用单一框架解释一切并发现这是逃避主义、用一个类比再否定它换更准的、识别 AI 输出里推论冒充观察、把情绪着色和情绪驱动分开。

做完这件事之后，回头看——这些动作有一个共同结构。它们都是**对正在发生的思维过程本身的实时监控**。不是想完之后回头看，是想的同时还有一层在看。

这件事被很多人称为"元认知"——但市面上关于元认知的内容要么是学术综述（读不下去），要么是鸡汤清单（读完没法执行）。两者之间的缝隙——**具体的、可调用的、和真实工作样本对应的动作清单**——是空的。

这份 skill 填这个缝隙。

## 一个发现

做完抽取后我发现一件事——这些动作如果让 AI 来执行，可能比我自己执行更稳定。

原因是**调用悖论**：需要元认知才能想起做元认知。一个有这种带宽的人不需要清单（动作已经在跑），一个没有这种带宽的人记不住要用清单（因为这件事本身需要元认知触发）。中间的位置——有带宽但希望系统化、节省调用成本——清单对这种人有用。

但更有意思的是——把清单作为指令给 AI，让 AI 在用户的每次思考过程中**自动启动检查**——这绕过了调用悖论。用户不需要记得用清单，AI 总在那里。

这个发现是这份 repo 存在的核心理由。它不是另一份"如何提高元认知"的文章，它是一个**让 AI 接管元认知监控工作**的具体工具。

## 一个立场

这份 skill 有明确的立场——

**真实判断 > 舒适感**。元认知工作会让你不舒服，因为它要求你回去拆已经写好的想法。如果你优先要顺畅愉悦的思考体验，不要用这份 skill。

**承认错误 > 保持一致**。元认知的核心动作之一是识别自己之前的判断有问题。一个把"保持一致"看得比"接近真实"更重的人，做不了这件事，用这份 skill 也不会有效。

**复杂性 > 单一框架**。这份 skill 持续抵抗单一框架吞掉多维现象。如果你想要一个解释一切的体系，这份 skill 会持续打断你。

**慢 > 快**。元认知动作消耗时间。如果你在一个奖励快速反应的环境里、需要持续快速输出，启动元认知检查会让你显得迟钝。这是真实的代价，要权衡。

我把这些立场说出来，是因为如果你不接受——这份 skill 对你不会有效。骗你说它对所有人都好是不诚实的。

## 一个限制

最后说一件这份 skill 做不到的事。

它**不能让一个底层带宽不足的人变得带宽充足**。原因不是动机或努力问题，是认知架构的差异。

让我尝试解释这是什么意思——

"带宽"指的是在一个意识时刻里能同时持有几件事。一个高带宽的人能同时持有"我之前说过 X"+"我现在想说 Y"+"X 和 Y 矛盾"+"矛盾需要解释"——四件事在场，然后做出回应。

一个低带宽的人不能持有这么多事。当 ta 说 Y 的时候，X 已经从可访问区退出了。不是 ta "忘了"或者"装作没说过"——是 X 这个信息在 ta 的系统里此刻不存在。所以 ta 说 Y 时是真诚的，没有矛盾感，因为矛盾需要两个东西同时在场才能产生。

这种差异不是知识差异。教一个低带宽的人"应该跨时刻一致"——ta 理解这个原则，但执行不了，因为执行需要在说 Y 之前能 access X。系统层面 access 不到，原则层面再清楚也没用。

加载这份 skill 给一个低带宽的人——AI 提出的检查问题会被 ta 系统的 self-protection 进程处理为"这是攻击"或"这是干扰"，而不是真的回去拆自己的想法。Skill 进得去，但被改造了。

所以诚实地说——这份 skill 对一些人很有用（已经有带宽，需要系统化的人），对另一些人没用（带宽不足，需要的不是 skill 是别的东西），对极少数人有害（用 skill 来表演元认知，让自己感觉更聪明）。

试用之后判断你属于哪一类。

## 关于继续改进

这份 skill 是 v1.0。可能有些动作我漏了、有些动作其实没必要、有些表述可以更准确。

如果你用了一段时间发现某些动作对你特别有用、某些根本调用不起来、或者发现新的动作——通过 GitHub issues 或 PR 告诉我，或者 fork 一个自己的版本。

这件事不是封闭系统。它本来就是从一个人的真实工作里抽出来的——别人的真实工作里会有别的动作，这些动作合起来才是更完整的 skill 库。

CC BY-SA 4.0 协议设计就是为了这种集体改进——你可以拿走、改写、再分享，只要署名并保持开放。
